# Req 命令

将模糊的功能想法转化为清晰、结构化的产品需求文档（PRD）。**此命令只生成 PRD 文档，从不执行代码或规划。**

> **调用**：此命令自动调用 `requirement-analyst` agent。

## ⚠️ 重要说明

**`/req` 只生成 PRD 文档，绝不执行：**
- 不创建实现规划（使用 `/plan`）
- 不编写代码（使用 `/tdd`）
- 不执行任何操作

**`/req` 完成后，你可以选择：**
1. 运行 `/plan` —— 基于 PRD 创建详细实施计划
2. 运行 `/tdd` —— 直接进入测试驱动开发
3. 直接告诉我 —— "根据 PRD 执行"，我将立即开始实现
4. 继续讨论需求 —— 修改 PRD 或补充需求

## 此命令的功能

1. **澄清需求** - 通过结构化问题引导你
2. **调研分析** - 调研现有代码库、模式和解决方案
3. **结构化思考** - 将高层需求分解为具体功能
4. **生成 PRD** - 生成标准化的 PRD 文档

## 使用时机

使用 `/req`：
- 想法模糊但不确定具体细节
- 需求只存在于脑海中
- 需要验证范围后再评估工作量
- 准备文档以便团队对齐

**跳过 `/req`**，直接使用 `/plan`：
- 需求已充分文档化
- 简单的 bug 修复或单行改动
- 已与相关方讨论过细节

## PRD 自动保存

生成的 PRD 将自动保存到项目目录：

**保存路径：** `.claude/reqs/YYYYMMDD-feature-name.md`

**文件名规则：**
- 日期前缀：当前日期（YYYYMMDD）
- 功能名称：从需求中提取的核心功能关键词
- 示例：`20250406-user-auth-system.md`

**好处：**
- 自动归档所有需求文档
- 便于后续查阅和管理
- 与代码变更历史对应

## PRD 输出格式

生成的 PRD 遵循以下结构：

1. **概述** - 2-3 句话总结
2. **调研发现** - 现有解决方案、技术现状、需要进一步调研的内容（可选但推荐）
3. **目标与成功标准** - 可衡量的目标
4. **用户故事** - 作为[用户]，我想要[目标]，以便[收益]
5. **功能需求** - 具体能力
6. **非功能需求** - 性能、安全、用户体验
7. **约束与假设** - 限制和前提条件
8. **待解决问题** - 需要解决的疑问
9. **下一步** - 提示使用 `/plan` 或 `/tdd`

## 与其他命令的集成

`/req` 生成 PRD 后，你有三个选择：

### 选项 1：创建详细规划
运行 `/plan`，agent 将：
- 分析 PRD 并识别风险
- 制定分阶段的实施计划
- 等待你确认后再执行

### 选项 2：直接进入开发
运行 `/tdd`，agent 将：
- 基于 PRD 直接开始测试驱动开发
- 先写测试，再实现，再重构
- 自动验证 80%+ 覆盖率

### 选项 3：直接执行
直接说 **"根据 PRD 执行"**，我将：
- 理解 PRD 内容
- 立即开始实现
- 无需额外的规划步骤

### 选项 4：继续讨论需求
直接说 **"修改 PRD"** 或 **"补充需求"**，我将：
- 继续和你讨论需求细节
- 根据你的反馈修改 PRD
- 迭代直到需求清晰完整

**注意**：这四个选项完全由你选择，`/req` 不会自动调用任何命令。

## 最佳实践

1. **坦诚表达不确定性** - 不确定的地方直接说明
2. **充分回答问题** - 更多上下文 = 更好的 PRD
3. **质疑假设** - agent 会质疑看似不清楚的地方
4. **迭代优化** - 可以多次修改 PRD
5. **规划前审阅** - 确保 PRD 完整后再运行 `/plan`
