# /learn-eval - 提取、评估，然后保存

在编写任何技能文件之前，扩展 `/learn` 添加质量门控、保存位置决策和知识放置感知。

## 提取内容

寻找：

1. **错误解决模式** — 根本原因 + 修复 + 可复用性
2. **调试技巧** — 非显而易见的步骤、工具组合
3. **变通方案** — 库的特殊行为、API 限制、版本特定的修复
4. **项目特定模式** — 约定、架构决策、集成模式

## 流程

1. 审查会话以寻找可提取的模式
2. 识别最有价值/可复用的洞见

3. **确定保存位置：**
   - 询问："这个模式在其他项目中有用吗？"
   - **全局** (`~/.claude/skills/learned/`)：可在 2+ 个项目中使用的通用模式（bash 兼容性、LLM API 行为、调试技巧等）
   - **项目**（当前项目的 `.claude/skills/learned/`）：项目特定知识（特定配置文件的特性、项目特定的架构决策等）
   - 不确定时，选择全局（全局 → 项目比反向更容易）

4. 使用此格式起草技能文件：

```markdown
---
name: pattern-name
description: "少于 130 字符"
user-invocable: false
origin: auto-extracted
---

# [描述性模式名称]

**提取时间：** [日期]
**上下文：** [此模式适用场景的简要描述]

## 问题
[此模式解决的问题 - 具体说明]

## 解决方案
[模式/技巧/变通方案 - 带代码示例]

## 何时使用
[触发条件]
```

5. **质量门控 — 检查清单 + 整体评估**

   ### 5a. 必需检查清单（通过实际读取文件验证）

   在评估草案之前执行 **以下所有操作**：

   - [ ] 通过关键字 grep `~/.claude/skills/` 和相关项目 `.claude/skills/` 文件以检查内容重叠
   - [ ] 检查 MEMORY.md（项目和全局）以查找重叠
   - [ ] 考虑追加到现有技能是否足够
   - [ ] 确认这是可复用模式，不是一次性修复

   ### 5b. 整体评估

   综合检查清单结果和草案质量，然后选择 **以下之一**：

   | 评估结果 | 含义 | 下一步操作 |
   |---------|------|-----------|
   | **保存** | 独特、具体、范围明确 | 继续步骤 6 |
   | **改进后保存** | 有价值但需要完善 | 列出改进项 → 修订 → 重新评估（一次） |
   | **吸收到 [X]** | 应追加到现有技能 | 显示目标技能和添加内容 → 步骤 6 |
   | **放弃** | 琐碎、冗余或过于抽象 | 解释原因并停止 |

   **指导维度**（用于评估，非评分）：

   - **具体性与可操作性**：包含可立即使用的代码示例或命令
   - **范围匹配**：名称、触发条件和内容一致，聚焦于单一模式
   - **独特性**：提供现有技能未涵盖的价值（由检查清单结果决定）
   - **可复用性**：未来会话中存在真实的触发场景

6. **评估特定的确认流程**

   - **改进后保存**：呈现必需的改进项 + 修订后的草案 + 重新评估后的更新检查清单/评估；如果修订后的评估结果是 **保存**，在用户确认后保存，否则遵循新的评估
   - **保存**：呈现保存路径 + 检查清单结果 + 1 行评估理由 + 完整草案 → 用户确认后保存
   - **吸收到 [X]**：呈现目标路径 + 添加内容（diff 格式）+ 检查清单结果 + 评估理由 → 用户确认后追加
   - **放弃**：仅显示检查清单结果 + 理由（无需确认）

7. 保存 / 吸收到确定的位置

## 步骤 5 的输出格式

```
### 检查清单
- [x] skills/ grep: 无重叠（或：发现重叠 → 详情）
- [x] MEMORY.md: 无重叠（或：发现重叠 → 详情）
- [x] 现有技能追加：新文件合适（或：应追加到 [X]）
- [x] 可复用性：已确认（或：一次性 → 放弃）

### 评估结果：保存 / 改进后保存 / 吸收到 [X] / 放弃

**理由：**（解释评估结果的 1-2 句话）
```

## 设计原理

此版本用基于检查清单的整体评估系统替代了之前的 5 维度数字评分标准（具体性、可操作性、范围匹配、非冗余、覆盖范围，1-5 分评分）。现代前沿模型（Opus 4.6+）具有强大的情境判断能力 — 将丰富的定性信号强制转换为数字评分会失去细微差别并可能产生误导性总分。整体方法让模型自然地权衡所有因素，产生更准确的保存/放弃决策，而显式检查清单确保不会跳过关键检查。

## 注意事项

- 不要提取琐碎的修复（拼写错误、简单语法错误）
- 不要提取一次性问题（特定的 API 中断等）
- 专注于能在未来会话中节省时间的模式
- 保持技能聚焦 — 每个技能一个模式
- 当评估结果为吸收时，追加到现有技能而非创建新文件
