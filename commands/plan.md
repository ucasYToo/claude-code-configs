# Plan 命令

此命令调用 **planner** agent 在编写任何代码之前创建全面的实施计划。

## 此命令的功能

1. **重述需求** - 明确需要构建什么
2. **识别风险** - 暴露潜在问题和阻塞项
3. **创建步骤计划** - 将实施分解为多个阶段
4. **等待确认** - 必须获得用户批准后才能继续

## 何时使用

在以下情况使用 `/plan`：
- 开始新功能
- 进行重大架构变更
- 处理复杂重构
- 多个文件/组件将受影响
- 需求不明确或模糊

## 工作原理

planner agent 将：

1. **分析请求** 并用清晰的术语重述需求
2. **分解为阶段** 制定具体、可执行的步骤
3. **识别依赖** 组件之间的依赖关系
4. **评估风险** 和潜在阻塞项
5. **估算复杂度** （高/中/低）
6. **呈现计划** 并 **等待** 您的明确确认

## 使用示例

```
用户: /plan 我需要添加市场结算时的实时通知

Agent (planner):
# 实施计划：实时市场结算通知

## 需求重述
- 当用户关注的市场结算时发送通知
- 支持多种通知渠道（应用内、邮件、Webhook）
- 确保通知可靠送达
- 包含市场结果和用户持仓结果

## 实施阶段

### 阶段 1：数据库架构
- 添加 notifications 表：id, user_id, market_id, type, status, created_at
- 添加 user_notification_preferences 表用于渠道偏好
- 在 user_id 和 market_id 上创建索引以提高性能

### 阶段 2：通知服务
- 在 lib/notifications.ts 创建通知服务
- 使用 BullMQ/Redis 实现通知队列
- 添加失败重试逻辑
- 创建通知模板

### 阶段 3：集成点
- 接入市场结算逻辑（当状态变为"已结算"时）
- 查询市场中所有有持仓的用户
- 为每个用户入队通知

### 阶段 4：前端组件
- 在顶部创建 NotificationBell 组件
- 添加 NotificationList 弹窗
- 通过 Supabase 订阅实现实时更新
- 添加通知偏好设置页面

## 依赖项
- Redis（用于队列）
- 邮件服务（SendGrid/Resend）
- Supabase 实时订阅

## 风险
- 高：邮件送达率（需要 SPF/DKIM）
- 中：单个市场 1000+ 用户的性能
- 中：市场频繁结算导致的通知轰炸
- 低：实时订阅开销

## 预估复杂度：中
- 后端：4-6 小时
- 前端：3-4 小时
- 测试：2-3 小时
- 总计：9-13 小时

**等待确认**：继续执行此计划？(是/否/修改)
```

## 重要说明

**关键**：planner agent 在您明确用"是"或"继续"等肯定性回复确认之前 **不会** 编写任何代码。

如果您想要更改，请回复：
- "修改：[您的更改]"
- "不同方案：[替代方案]"
- "跳过阶段 2，先做阶段 3"

## 与其他命令的集成

计划完成后：
- 使用 `/tdd` 进行测试驱动开发
- 如果出现构建错误使用 `/build-fix`
- 使用 `/code-review` 审查完成的实现
