# 规划工作流

创建有效实施计划的可复用模式。在将复杂功能分解为可操作步骤时使用这些模式。

## 何时使用

- 开始新功能或项目
- 进行架构变更
- 规划复杂重构
- 将大任务分解为可管理的部分
- 估算工作量和识别风险

## 规划原则

### 1. 从需求开始
- 在规划如何构建之前理解需要构建什么
- 明确成功标准
- 识别约束和假设

### 2. 分阶段思考
- 将工作分解为可独立交付的块
- 每个阶段都应该自己提供价值
- 避免全有或全无的计划

### 3. 具体明确
- 使用确切的文件路径和函数名
- 定义清晰的输入和输出
- 指定预期行为

### 4. 考虑依赖关系
- 识别什么必须在什么之前完成
- 注意外部依赖（API、服务、团队）
- 规划集成点

## 计划结构

### 标准模板

```markdown
# 实施计划: [功能名称]

## 概述
[2-3 句话总结我们要构建什么以及为什么]

## 需求重述
- [需求 1]
- [需求 2]
- [成功标准]

## 架构变更
- [变更 1: 文件路径和描述]
- [变更 2: 文件路径和描述]

## 实施步骤

### 阶段 1: [阶段名称]
1. **[步骤名称]** (文件: path/to/file.ts)
   - 操作: 要采取的具体操作
   - 原因: 此步骤的理由
   - 依赖: 无 / 需要步骤 X
   - 风险: 低/中/高

### 阶段 2: [阶段名称]
...

## 测试策略
- 单元测试: [要测试的文件]
- 集成测试: [要测试的流程]
- E2E 测试: [要测试的用户旅程]

## 风险与缓解
- **风险**: [描述]
  - 缓解: [如何解决]

## 成功标准
- [ ] 标准 1
- [ ] 标准 2
```

## 阶段规划

### 阶段类型

| 阶段 | 目的 | 示例 |
|-------|---------|---------|
| **基础** | 设置基础设施 | 数据库架构、配置、工具 |
| **核心** | 实现主要功能 | API 端点、业务逻辑 |
| **集成** | 连接组件 | 前端与后端连接 |
| **完善** | 错误处理、边界情况 | 验证、错误消息 |
| **优化** | 性能、监控 | 缓存、日志、指标 |

### 阶段规模指南

- **小**: 1-2 个文件, < 2 小时
- **中**: 3-5 个文件, 2-4 小时
- **大**: 6+ 个文件, 4-8 小时（考虑拆分）

每个阶段应该是：
- 可独立审查
- 可隔离测试
- 可合并不破坏主干

## 风险评估

### 常见风险类别

| 类别 | 示例 | 缓解 |
|----------|----------|------------|
| **技术** | 新技术、复杂算法 | 先进行 spike/原型 |
| **集成** | 第三方 API、外部服务 | 尽早 mock 接口 |
| **性能** | 大数据量、实时 | 尽早基准测试，稍后优化 |
| **安全** | 认证、敏感数据 | 每个阶段进行安全审查 |
| **依赖** | 被其他团队/功能阻塞 | 并行工作、备用计划 |

### 风险登记模板

```markdown
## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解 |
|------|------------|--------|------------|
| API 速率限制 | 中 | 高 | 实施缓存、退避 |
| 数据库迁移失败 | 低 | 高 | 在 staging 测试、先备份 |
| 10k 用户的性能 | 中 | 中 | 在阶段 2 进行负载测试 |
```

## 规划检查清单

在最终确定计划之前，验证：

- [ ] 所有需求都已解决
- [ ] 每个步骤都有清晰的文件路径
- [ ] 依赖关系已识别
- [ ] 风险有缓解策略
- [ ] 测试方法已定义
- [ ] 阶段可以独立交付
- [ ] 成功标准是可衡量的
- [ ] 工作量估算是现实的

## 示例计划

```markdown
# 实施计划: Stripe 订阅计费

## 概述
添加具有免费/专业/企业层的订阅计费。用户通过
Stripe Checkout 升级，webhook 事件保持订阅状态同步。

## 需求
- 三层: 免费（默认）、专业版 ($29/月)、企业版 ($99/月)
- Stripe Checkout 用于支付流程
- Webhook 处理程序用于订阅生命周期事件
- 基于订阅层的功能门控

## 架构变更
- 新表: `subscriptions` (user_id, stripe_customer_id, stripe_subscription_id, status, tier)
- 新 API 路由: `app/api/checkout/route.ts`
- 新 API 路由: `app/api/webhooks/stripe/route.ts`
- 新中间件: 检查订阅层
- 新组件: `PricingTable`

## 实施步骤

### 阶段 1: 数据库与后端
1. **创建订阅迁移**
   - 文件: `supabase/migrations/004_subscriptions.sql`
   - 依赖: 无
   - 风险: 低

2. **创建 Stripe webhook 处理程序**
   - 文件: `src/app/api/webhooks/stripe/route.ts`
   - 依赖: 步骤 1
   - 风险: 高 — 验证签名

### 阶段 2: 结账流程
3. **创建结账 API 路由**
   - 文件: `src/app/api/checkout/route.ts`
   - 依赖: 步骤 1
   - 风险: 中

4. **构建定价页面**
   - 文件: `src/components/PricingTable.tsx`
   - 依赖: 步骤 3
   - 风险: 低

## 测试策略
- 单元: Webhook 解析、层检查
- 集成: 结账会话、webhook 处理
- E2E: 完整升级流程

## 风险与缓解
- **风险**: Webhook 事件乱序到达
  - 缓解: 使用时间戳、幂等更新

## 成功标准
- [ ] 用户可以通过 Stripe Checkout 升级
- [ ] Webhook 正确同步状态
- [ ] 免费用户被阻止访问专业功能
- [ ] 80%+ 测试覆盖率
```
