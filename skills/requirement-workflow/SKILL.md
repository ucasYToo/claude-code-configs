# 需求工作流

有效需求工程的可复用模式和技术。在引出、分析和记录产品需求时使用这些模式作为参考。

## 适用场景

- 开始新项目或功能
- 需求感觉模糊或不完整
- 利益相关者有冲突的理解
- 在投入资源之前需要验证范围
- 在投入设计之前验证范围
- 在设计新方案之前需要了解现有解决方案

## 渐进式澄清框架

需求澄清通常可分为三个阶段的问题集合。

### 阶段 1：核心问题
适用于任何需求澄清的起点：
1. **为谁**做这个？（目标用户/角色）
2. **解决什么**问题？（核心功能）
3. **为什么**重要？（业务/用户价值）

### 阶段 2：扩展分析
在核心问题明确后，可进一步探讨：
4. **何时**需要？（时间线、里程碑）
5. **约束** — 存在什么限制？（技术、预算、监管）
6. **范围** — 包含什么和不包含什么？

### 阶段 3：边界情况
对于关键或复杂功能，可深入挖掘：
7. **错误场景** — 异常情况如何处理？
8. **性能要求** — 响应时间、并发量？
9. **安全合规** — 数据隐私、访问控制？

## 5W 框架

| 问题 | 目的 | 示例答案 |
|------|------|----------|
| **谁** | 识别用户/角色 | "管理员用户"、"移动客户"、"API 集成商" |
| **什么** | 定义功能 | "导出报告"、"重置密码"、"同步数据" |
| **为什么** | 理解动机 | "合规要求"、"减少流失"、"与竞争对手持平" |
| **何时** | 设定时间线/里程碑 | "季度末"、"黑色星期五前"、"尽快" |
| **约束** | 暴露限制 | "必须使用现有认证"、"$10K 预算"、"GDPR 合规" |

## 积极倾听模式

当用户陈述需求时，深入探究：

| 用户说 | 用...探究 | 目标 |
|--------|-----------|------|
| "让它快" | "目标响应时间是多少？" | 获取具体指标 |
| "用户应该能够..." | "具体是哪些用户？" | 识别角色 |
| "支持所有浏览器" | "有什么特定版本必须支持吗？" | 定义范围 |
| "易于使用" | "什么任务应该在 < 3 次点击内完成？" | 使 UX 可操作化 |

## 研究阶段

### 研究目标

1. **现有产品能力盘点** — 当前系统是否已有相关功能？用户现在的 workaround 是什么？现有功能的边界在哪？
2. **用户场景验证** — 目标用户群体是否真有这个痛点？
3. **竞品/行业参考** — 类似产品通常怎么做？有什么常见坑？
4. **业务价值量化** — 这个功能能带来什么价值？有没有数据支撑？
5. **边界澄清** — 行业里类似功能的范围通常怎么划？哪些是标配，哪些是进阶？

### 研究方法

| 方法 | 工具 | 何时使用 |
|------|------|----------|
| **现有产品能力盘点** | `Read`, `Glob`, `Grep` | 查看产品文档、README、现有功能模块，了解当前能力边界 |
| **用户场景研究** | Web 搜索 | 了解目标用户的真实痛点和使用习惯 |
| **竞品分析** | Web 搜索、Web fetch | 从类似产品学习功能设计和范围边界 |
| **行业最佳实践** | Web 搜索、博客 | 了解常见做法和常见陷阱 |

**不研究的内容：** 代码库、技术选型、架构模式、库/框架对比。

### 研究检查清单

- [ ] **现有产品能力** — 当前系统是否已有相关功能？现有功能的边界在哪？用户现在的 workaround 是什么？
- [ ] **用户验证** — 目标用户群体是否确认有这个需求？
- [ ] **竞品对标** — 类似产品通常提供什么范围？超出范围的常见做法是什么？
- [ ] **行业陷阱** — 这个功能最常见的失败场景或用户投诉是什么？
- [ ] **业务约束** — 有没有法规、合规、预算、时间线等限制？
- [ ] **成功指标** — 行业里通常用什么指标衡量这个功能的成功？

### 记录研究发现

在 PRD 中可参考以下结构：

```markdown
## 2. 研究发现

### 2.1 现有产品能力
- **当前能力**: 系统已支持基础数据列表展示，但无导出功能
- **用户 workaround**: 手动复制粘贴到 Excel，耗时 30 分钟/次
- **功能边界**: 现有列表支持筛选和排序，但筛选条件无法保存
- **复用机会**: 筛选逻辑可与导出功能共用

### 2.2 用户场景验证
- **目标用户**: 运营人员，每天需要导出数据做周报
- **痛点强度**: 高频（每周）、高耗时、易出错

### 2.3 竞品参考
- **竞品 A**: 支持 CSV/Excel 导出，可筛选字段，但无法定时导出
- **竞品 B**: 支持定时邮件发送，但格式固定
- **行业标配**: 筛选 + 即时导出为最低配；定时导出为进阶功能

### 2.4 常见陷阱
- 用户常误以为"导出全部数据"是默认需求，实际 90% 场景只需要筛选后导出
- 权限控制容易被遗漏：谁可以导什么数据

### 2.5 需要进一步确认
- [ ] 与运营团队确认导出频率和格式偏好
- [ ] 确认数据权限矩阵
```

## PRD 模板适用场景

本 skill 提供三种模板，各自适用于不同的需求复杂度：

- **`templates/prd-quick.md`** — 适合边界清晰、涉及文件少、可在一天内完成的简单功能。
- **`templates/prd-full.md`** — 适合涉及多个用户场景、需要分模块交付、有一定研究价值的标准功能。
- **`templates/prd-ai-product.md`** — 适合 AI 驱动功能或需要深入考虑模型选择、安全合规、用户体验的复杂产品。

## 用户故事模板

### 标准格式
```
作为一个 [用户类型]，我想要 [目标]，以便 [收益]。
```

### 带验收标准
```
作为一个 [用户类型]，我想要 [目标]，以便 [收益]。

**验收标准：**
- [ ] 标准 1（given/when/then 或特定条件）
- [ ] 标准 2
- [ ] 标准 3

**优先级：** 必须有 / 应该有 / 可以有
**估算：** 小 / 中 / 大
```

### 常见用户故事模式

| 模式 | 示例 |
|------|------|
| **认证** | 作为一个用户，我想要用我的邮箱登录，以便我可以安全地访问我的账户。 |
| **数据输入** | 作为一个管理员，我想要通过 CSV 批量上传用户，以便我不必一个一个添加他们。 |
| **通知** | 作为一个客户，我想要在订单发货时接收邮件提醒，以便我知道何时可以收货。 |
| **报告** | 作为一个经理，我想要查看每日销售报告，以便我可以跟踪团队绩效。 |
| **集成** | 作为一个开发者，我想要 REST API 访问，以便我可以与我们的内部工具集成。 |

## 验收标准模式

### 模式 1：Given/When/Then（BDD 风格）
```
Given [前置条件]
When [操作]
Then [预期结果]
```

示例：
```
Given 用户在登录页面
When 他们输入有效凭证
Then 他们被重定向到仪表板
And 他们的会话令牌被存储
```

### 模式 2：检查清单风格
```
- [ ] 用户可以输入邮箱和密码
- [ ] 系统根据数据库验证凭证
- [ ] 无效凭证显示错误消息 "邮箱或密码无效"
- [ ] 成功登录重定向到 /dashboard
- [ ] 会话在 24 小时不活动后过期
```

### 模式 3：约束风格（非功能需求）
```
- [ ] 第 95 百分位响应时间 < 200ms
- [ ] 支持 10,000 并发用户
- [ ] 适用于 Chrome、Firefox、Safari（最近 2 个版本）
- [ ] 符合 WCAG 2.1 AA 可访问性标准
```

## 约束分类

### 技术约束
| 类别 | 示例 |
|------|------|
| **平台** | "必须在 iOS 14+ 上运行"、"仅限无服务器架构" |
| **集成** | "必须使用现有的 Salesforce 连接"、"使用 Stripe 支付" |
| **性能** | "页面加载 < 3s"、"API 延迟 < 100ms" |
| **安全** | "符合 SOC2"、"数据必须留在欧盟" |

### 业务约束
| 类别 | 示例 |
|------|------|
| **预算** | "最多 $50K"、"使用现有许可证" |
| **时间线** | "在假期季节前"、"季度末" |
| **资源** | "只有 2 个开发者可用"、"没有设计支持" |
| **合规** | "符合 GDPR"、"需要 HIPAA" |

### 用户约束
| 类别 | 示例 |
|------|------|
| **可访问性** | "WCAG 2.1 AA"、"兼容屏幕阅读器" |
| **本地化** | "支持 12 种语言"、"RTL 布局" |
| **设备** | "在 320px 宽度上工作"、"触摸友好" |

## 红旗检查清单

在需求收集期间注意这些警告信号：

- [ ] 模糊术语："用户友好"、"快"、"简单"、"直观"
- [ ] 没有定义成功指标
- [ ] 缺少用户上下文（这是给谁用的？）
- [ ] 未说明的假设
- [ ] 范围蔓延指标（"还有..."、"如果我们添加..."）
- [ ] 冲突的需求
- [ ] 没有时间线或里程碑
- [ ] 缺少错误场景
- [ ] 没有识别约束
- [ ] 解决方案伪装成需求（"前端使用 React"）

## 输出规范

### 文件命名约定

PRD 文件统一保存至：
```
.claude/reqs/YYYYMMDD-feature-name.md
```

**命名规则：**
1. **提取功能名称**：将用户描述的核心功能转为英文 slug（例如，"user authentication system" → "user-auth-system"）
2. **生成文件名**：当前日期 + 功能名称 slug
3. **检查重复**：如果文件已存在，添加数字后缀：`YYYYMMDD-feature-name-1.md`
4. **确保目录存在**：如 `.claude/reqs/` 不存在则创建

**示例：**
- 用户描述："I need a user authentication system"
- 生成文件名：`.claude/reqs/20250409-user-auth-system.md`

### PRD 内容要求

在最终确定 PRD 之前，验证：

#### 内容质量
- [ ] 每个需求都是具体且明确的
- [ ] 每个用户故事都有清晰的验收标准（描述"什么结果是对的"，不涉及"怎么实现"）
- [ ] 成功标准是可衡量的
- [ ] 约束已记录（业务/用户层面，不写技术约束）
- [ ] 考虑了边界情况和失败场景
- [ ] 明确列出范围边界（不做什么）
- [ ] 为每个故事分配了优先级
- [ ] 识别了需求间的依赖关系
- [ ] 未决问题有负责人
- [ ] **不包含技术实现细节**（无技术选型、架构方案、实现策略）

#### 研究质量
- [ ] 验证了用户场景的真实性（非假设）
- [ ] 参考了竞品/行业最佳实践
- [ ] 识别了常见陷阱和失败模式
- [ ] 量化了业务价值（如有数据）
- [ ] 如果需要，识别了进一步确认的事项
