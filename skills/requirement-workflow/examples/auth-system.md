# PRD: 用户认证系统

版本: v1.0.0 | 日期: 2026-04-06 | 负责人: 产品团队

---

## 1. 概述

实现安全的邮箱/密码认证系统，支持可选的 Google OAuth 登录，以替换安全审计标记的当前不安全系统。目标是消除明文密码存储，将账户接管事件减少 90%。

---

## 2. 研究发现

### 2.1 现有解决方案
- **当前代码库**: `src/auth/` 中的基本基于会话的认证，密码明文存储，缺少速率限制
- **行业标准**: 带刷新令牌轮换的 JWT
- **竞争分析**: 大多数使用 OAuth 优先，邮箱/密码回退
- **经验教训**: 现有实现缺少刷新令牌轮换和 MFA 支持

### 2.2 技术环境
- **考虑的选项**:
  - **Auth0**: 企业级，$500+/月，快速实施， vendor lock-in
  - **NextAuth.js**: 免费，Next.js 原生，良好社区支持
  - **自定义 JWT**: 完全控制，更高的维护负担
- **推荐方法**: NextAuth.js 用于速度和灵活性的平衡

### 2.3 需要进一步研究
- [ ] 与安全团队评估 NextAuth.js 与自定义解决方案
- [ ] 与产品团队确认 OAuth 提供商（Google？GitHub？其他？）
- [ ] 评估 MFA/2FA 的需求和时间线

## 4. 用户故事

### 4.1 邮箱/密码注册

**作为一个** 新客户，
**我想要** 用邮箱和密码注册，
**以便** 我可以在没有第三方依赖的情况下创建账户。

**验收标准：**
- [ ] Given 用户在注册页面，When 输入有效邮箱和密码，Then 账户创建成功
- [ ] Given 用户输入邮箱，When 邮箱格式无效，Then 显示 "请输入有效的邮箱地址"
- [ ] Given 用户输入密码，When 密码少于 12 位，Then 显示 "密码至少需要 12 个字符"
- [ ] Given 用户输入已存在的邮箱，When 提交注册，Then 显示 "该邮箱已被注册"
- [ ] Given 注册成功，When 30 秒内，Then 用户收到验证邮件
- [ ] Given 用户未验证邮箱，When 尝试登录，Then 提示 "请先验证邮箱"

**优先级:** 必须有
**估算:** 中

### 4.2 社交登录

**作为一个** 客户，
**我想要** 用我的 Google 账户登录，
**以便** 我不需要记住另一个密码。

**验收标准：**
- [ ] Given 用户在登录页点击 "使用 Google 登录"，When 授权成功，Then 自动登录
- [ ] Given 用户首次使用 Google 登录，When 成功授权，Then 自动创建账户
- [ ] Given 用户已有邮箱账户，When 使用相同邮箱的 Google 登录，Then 提示关联账户
- [ ] Given 用户在账户设置中，When 点击 "取消关联 Google"，Then 解除 OAuth 关联

**优先级:** 应该有
**估算:** 中

## 7. 约束与假设

### 约束
- **技术约束**: 必须使用现有的 PostgreSQL 用户数据库
- **安全约束**: 密码必须使用 bcrypt 哈希
- **合规约束**: GDPR 合规的数据处理
- **预算约束**: 认证服务预算 $5K

### 假设
- **假设 1**: 用户有基本的邮箱访问能力
- **假设 2**: 大多数用户使用主流浏览器（Chrome/Firefox/Safari 最近 2 版本）
