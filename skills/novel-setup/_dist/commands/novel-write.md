# Novel Write - 章节写作

七人协作完成单章3000字正文创作。

## 七人协作流程

```
阶段1: 准备（并行）
  ├─ novel-research-collector 搜集本章资料
  └─ novel-lorekeeper 提供当前数据

阶段2: 创作规范
  novel-editor → 输出创作规范（场景表/骨架/字数分配）

阶段3: 写作
  novel-writer → 骨架 → 多Agent讨论（3人并行） → Editor综合决策 → 扩写3000+字正文

阶段4: 并行检查（3人同时）
  ├─ novel-character-growth-tracker: 成长分析（顾问，只提建议）
  ├─ novel-reader-proxy: 读者体验（顾问，只提建议）
  └─ novel-proofreader: 质量检查（唯一门禁）

阶段5: 综合决策
  novel-editor → 读取3份报告，判断是否通过

阶段6: 收尾
  novel-lorekeeper 更新数据
```

## 任务追踪要求

执行流程开始时，创建以下任务列表：
1. 阶段1: 资料搜集
2. 阶段2: 创作规范
3. 阶段3: 骨架创作与讨论
4. 阶段4: 并行质量检查
5. 阶段5: 综合决策
6. 阶段6: 数据更新与收尾

每个阶段开始时，将该任务标记为进行中；阶段完成后（或门禁通过后），标记为已完成。如门禁不通过需重写，保持当前任务为进行中状态并附加说明。

## 详细流程

### 1. Research-Collector 资料搜集（按需）

**工作流程**：
1. 读取本章大纲，识别场景和专业背景需求
2. 判断是否需要额外资料：
   - 涉及历史事实、专业领域、具体数据 → 需要搜集
   - 纯虚构场景、日常对话、已搜集过的背景 → 无需搜集
3. 如需搜集：搜索并保存到 `03-素材库/参考资料/第X章/`
4. 输出资料摘要（含关键事实清单）

### 2. Editor 制定创作规范

**必须读取**：
- 本章大纲
- `03-素材库/参考资料/第X章/`（Research-Collector 生成的资料）
- `05-数据/`（Lorekeeper 提供的人物/时间线数据）

**输出《第X章创作规范》**：
- 场景设计（地点/人物/目的）
- 高潮点分布
- 骨架规划（每段30-50字核心内容）
- 字数分配
- 【关键事实】（来自资料，Writer 必须遵守）
- 约束条件（必须包含/避免的内容）

### 3. Writer 创作

**Step 1: 输出骨架**
根据Editor规范，输出12+段落骨架，每段带扩写标记。

**Step 2: 多Agent骨架讨论（并行）**

**工具提示**：使用 `Agent` 工具**并行启动**以下3个Agent进行骨架评估。

Writer 输出骨架后，3个Agent并行评估：

**Reader-Proxy（顾问）**：
- 评估骨架爽点铺垫是否充分
- 评估章尾钩子吸引力
- 识别可能让读者弃文的隐患
- 输出读者视角建议

**Growth-Tracker（顾问）**：
- 评估主角成长节奏合理性
- 检查人物关系发展是否自然
- 验证性格转变是否有铺垫
- 输出成长曲线建议

**Proofreader（逻辑检查）**：
- 检查骨架与大纲一致性
- 识别逻辑漏洞和设定冲突
- 验证与前章数据连续性
- 输出逻辑检查报告

**Step 3: Editor综合决策**

Editor读取3份报告，做出决策：

| 情况 | Editor决策 | 后续动作 |
|------|-----------|----------|
| 无重大问题 | ✅ 通过 | 通知Writer直接扩写 |
| 有优化建议 | ⚠️ 优化后通过 | 整合建议，Writer调整骨架后扩写 |
| 有重大缺陷 | ❌ 打回 | Writer重写骨架，重新讨论 |

**如需交叉讨论**：如3个Agent的意见有重大分歧且难以裁定，Editor可触发 `/novel meeting` 进行多轮辩论，议题如"第X章骨架争议点讨论"。

**Step 4: 扩写正文（含实时资料查询）**

决策通过后，Writer扩写为3000+字正文。参考来源（按优先级）：

**文风参考**（关键！）：
1. `00-元数据/文字风格.md` 中的【参考来源配置】
2. 配置的外部参考文本（其他书籍/散文/教材）
3. 本书前3章（作为一致性基准）

**内容参考**：
- Editor 规范中的【关键事实】
- 本章资料（`03-素材库/参考资料/第X章/`）
- Agent讨论建议（如有）

**扩写策略**：
根据场景类型选择对应扩写示例（详见 `文字风格.md`）：
- 都市对话 → 示例A
- 动物/特殊视角 → 示例B
- 情感冲突 → 示例C
- 日常温馨 → 示例D
- 体育竞技 → 示例E

**反AI检测扩写检查点（Writer必须执行）**：

每完成3个段落，Writer必须自检：
1. **画面感检查**：这段如果拍成电影，画面是什么？
2. **生活化比喻检查**：用了什么生活化比喻？（饺子皮、飞机、蜂蜜面包）
3. **数字检查**：有具体数字吗？（48小时、9分男、三百多号人）
4. **吐槽密度检查**：喜剧段落有5-8句吐槽吗？
5. **句式多样性检查**：有长短句混合吗？有倒装/插入语吗？

如检查不通过，立即修改后再继续扩写。

**实时资料查询机制**：
Writer 扩写中遇到不确定的事实时：
1. 标记【待查：XXX】
2. 调用 Research-Collector 查询
3. 获取结果后继续扩写

示例：
- 【待查：2003年NBA工资帽】
- 【待查：CBA选秀报名流程】
- 【待查：当年大学生月生活费水平】

### 4. 并行检查（3人同时启动）

**工具提示**：使用 `Agent` 工具**并行启动**以下3个Agent，每个Agent独立调用。

**Growth-Tracker（顾问）**：
- 分析主角能力成长曲线
- 检查性格转变合理性
- 评估关系发展进度
- 输出成长建议报告（不阻塞流程）

**Reader-Proxy（顾问）**：
- 评估爽点效果和强度
- 检测追更动力（章尾钩子）
- 识别弃文风险（水文/毒点）
- **评估吐槽密度和喜剧效果**
- 输出读者体验报告（不阻塞流程）

**Proofreader（唯一门禁）**：
- 设定一致性检查
- 逻辑合理性检查
- **风格统一性检查**（对照 `文字风格.md` 中的参考配置）
- **反AI检测检查**（关键！）：
  - 情绪标点化使用（感叹号、重复）
  - 具体数字代替笼统描述
  - 旁观者视角（内心OS打分）
  - 极简短句塑造性格
  - 组合信号写动作
  - 生活化比喻（饺子皮、飞机耳等）
  - 电影镜头式画面感
  - 无AI高频连接词
  - 吐槽密度达标
- 原创性检查
- **AI检测风险评估**（低/中/高）
- 输出质量评分（≥6通过，<6打回）

### 5. Editor 综合决策

读取3份检查报告，做出最终决策：

| Proofreader评分 | AI检测风险 | 其他Agent建议 | Editor决策 |
|-----------------|------------|---------------|------------|
| ≥ 7分 | 低 | 无重大建议 | ✅ 通过 |
| ≥ 7分 | 低 | 有重要建议 | ⚠️ 通过，附参考建议 |
| 6-7分 | 中 | - | ⚠️ 通过，建议Writer优化 |
| < 6分 或 CRITICAL问题 | 任意 | - | ❌ 打回Writer重写 |
| 任意 | 高 | - | ❌ 打回Writer重写，必须降低AI检测风险 |

**AI检测高风险指征**（必须打回）：
- 连续使用AI高频连接词（此外/然而/综上所述）
- 缺乏具体数字，大量使用笼统描述
- 无生活化比喻，画面感薄弱
- 句式过于规整，缺少长短句变化
- 无情绪标点化，语气过于平稳
- 配角无标志性短句，工具化严重

### 6. 收尾（Lorekeeper 执行）

**更新数据文件**：
- characters.json（人物状态）
- timeline.json（时间线）
- foreshadowings.json（伏笔状态）
- 生成章节简介

**更新项目级文档**（关键！）：

**1. 更新 README.md**（项目根目录）
- **当前进度**: 更新为最新章节号
- **总字数**: 累加本章字数
- **最近更新**: 添加本章完成记录
```markdown
| 日期 | 章节 | 备注 |
|------|------|------|
| 2026-03-21 | 第11章 | 主角觉醒系统 |
| 2026-03-20 | 第10章 | 宿舍冲突 | ← 原有记录保留
```

**2. 更新 CLAUDE.md**（项目根目录）
- **当前进度**: 更新已完成的章节范围
- **总字数**: 更新累计字数
- **下章计划**: 更新为下一章信息（从大纲读取）
- **近期伏笔状态**: 更新本章处理/埋下的伏笔

**更新时机**：每章通过 Proofreader 检查后立即更新

**注意**：README.md 和 CLAUDE.md 是项目级文档，每次更新追加新记录，保留历史

## 使用方式

```
/novel write 11                    # 写第11章（标准模式）
/novel write 11 --fast             # 快速模式（跳过顾问检查）
/novel write 11 --urgent           # 紧急模式（Editor→Writer→Proofreader）
```

## 输出文件

### 章节文件
- `04-正文/第X篇-XXX/第XXX章.md` - 正文
- `04-正文/第X篇-XXX/第XXX章-简介.md` - 章节简介

### 数据文件（自动更新）
- `05-数据/characters.json` - 人物状态
- `05-数据/timeline.json` - 时间线
- `05-数据/foreshadowings.json` - 伏笔状态

### 项目文档（自动更新）
- `README.md` - 更新进度、字数、最近更新记录
- `CLAUDE.md` - 更新当前进度、下章计划、伏笔状态

## 纪律要求

1. **每章必须3000字以上**
2. **必须经过多Agent骨架讨论后才能扩写**
3. **Proofreader检查通过才算完成**
4. **每章必须更新Lorekeeper数据**
5. **章尾必须有钩子**（Editor在创作规范中设计）
6. **必须通过反AI检测**（关键！）：
   - 笑点规划达标（2-3个喜剧释放点）
   - 画面感检查通过（每段都有电影镜头式描写）
   - 吐槽密度达标（20-30%占比）
   - 配角三句话法则执行
   - 具体数字使用到位
   - 无AI高频连接词
   - AI检测风险评估为"低"
