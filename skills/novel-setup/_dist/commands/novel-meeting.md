# Novel Meeting - 多Agent会议

针对需要多方讨论的事项（大纲设计、人物决策、剧情方向、章节审阅），召集相关Agent进行讨论。

## 典型用法

```
/novel meeting "推理大纲结构"            # 大纲设计
/novel meeting "审阅第11章"              # 章节审阅
/novel meeting "第50章主角应该输还是赢"  # 剧情决策
/novel meeting "女主感情线推进时机"      # 人物设定讨论
/novel meeting "分析前3章文风"           # 分析本书已有章节风格
/novel meeting "分析《猫猫》文风并导入"   # 分析外部书籍（如你的另一本书）
/novel meeting "对比当前风格与参考差异"   # 风格一致性检查
/novel meeting "优化心理描写风格"         # 针对特定风格要素优化
```

## 议题自动匹配

根据议题关键词，自动匹配参与Agent：

| 议题类型 | 关键词 | 参与Agent |
|----------|--------|-----------|
| 剧情决策 | 输/赢/战/对抗/第X章 | Editor + Reader-Proxy + Growth-Tracker + Writer |
| 人物设定 | 人物/关系/感情/CP | Editor + Growth-Tracker + Lorekeeper + Writer |
| 世界观设定 | 设定/规则/背景/资料 | Editor + Proofreader + Lorekeeper + Research-Collector |
| 写作策略 | 怎么写/文风/手法 | Editor + Writer + Reader-Proxy + Proofreader |
| 文风分析 | 文风/风格/分析/导入 | Editor + Writer + Proofreader |
| 综合/其他 | - | 全员7人 |

## 任务追踪要求

会议开始时，创建以下任务列表：
1. Step 1: Editor确认议题
2. Step 2: 第一轮并行输出
3. Step 3: 第二轮交叉讨论
4. Step 4: Editor整合决策
5. Step 5: 更新项目文档

每个步骤开始时，将该任务标记为进行中；步骤完成后，标记为已完成。

## 会议流程（多轮辩论）

### Step 1: Editor确认议题
- 明确讨论范围和决策类型
- 收集背景信息（读取相关文件）
- 确定优先考量

### Step 2: 第一轮 - 并行输出

**工具提示**：使用 `Agent` 工具**并行启动**所有匹配的参与Agent（每个Agent一个独立的`Agent`调用）。

参与Agent各自输出详细分析（300字+）：

**常规Agent（Editor/Writer/Growth-Tracker/Reader-Proxy/Proofreader/Lorekeeper）**：
- 立场/判断
- 核心理由（3点，带具体依据）
- 具体建议（2-3条，可操作）

**Research-Collector（如匹配"世界观/资料"类议题）**：
- 搜索与议题相关的背景资料
- 输出关键事实和数据（带来源）
- 标注可直接使用的【创作素材】
- 保存资料到 `03-素材库/参考资料/会议-[议题]/`

### Step 3: 第二轮 - 交叉讨论（真正的多轮讨论）

**关键**：这不是简单的再次调用，而是要把第一轮的所有观点汇总后，作为上下文传递给每个Agent。

**实现步骤**：

1. **汇总第一轮输出**（由主流程完成）：
   ```markdown
   ## 第一轮观点汇总

   ### @Editor
   [Editor的第一轮观点摘要]

   ### @Writer
   [Writer的第一轮观点摘要]

 ### @Reader-Proxy
   [Reader-Proxy的第一轮观点摘要]

   ...（所有参与Agent）
   ```

2. **并行启动第二轮讨论**：
   使用 `Agent` 工具**再次并行启动**所有参与Agent，**在prompt中传入完整的《第一轮观点汇总》**，并明确要求：
   - 阅读其他Agent的观点
   - 使用 `@AgentName` 格式引用回应
   - 标记 **【同意】** / **【反对】** / **【补充】**

**第二轮输出格式**：
```markdown
## 交叉讨论 - [AgentName]

### 对其他Agent的回应

@AgentA 关于XXX的观点：
【同意】理由：XXX

@AgentB 关于YYY的建议：
【反对】理由：XXX
【补充】建议改为：XXX

### 修正/补充我的观点
基于讨论，我修正之前的观点：XXX
```

**辩论格式示例**：
```markdown
@Character-Growth-Tracker 关于主角能力提升的建议：
【同意】建议放缓提升速度，从读者角度补充：
如果中投从40直接跳到60，爽感确实强，但会削弱后续成长空间。
建议改为45-50，并增加训练过程描写。

@Writer 这个调整在写作上可行吗？会增加多少字数？
```

### Step 4: 第三轮 - 收敛共识（如需要）

如第二轮仍有重大分歧，进行第三轮简短回应（各100字内）。

**实现步骤**：
1. 汇总第二轮的所有讨论内容
2. 识别仍有分歧的议题
3. 使用 `Agent` 工具**并行启动**有分歧的Agent
4. 在prompt中传入完整的讨论历史
5. 要求Agent使用 **【最终立场】** 标记最终观点

### Step 5: Editor整合决策报告
- 综合所有Agent意见
- 如有分歧，说明裁定理由
- 输出可执行的决策清单

### Step 6: 更新项目文档（如涉及设定变更）

**需要更新 CLAUDE.md 的情况**：
- 核心设定变更 → 更新【核心设定】部分
- 主角/重要人物设定变更 → 更新【主角设定】或【重要配角】
- 新增伏笔 → 更新【近期伏笔状态】
- 下章计划变更 → 更新【下章计划】

**需要更新 README.md 的情况**：
- 大纲结构调整 → 更新【计划章节】
- 项目状态变更 → 更新【状态】

**更新方式**：
- Editor 或 Lorekeeper 根据会议决议，使用 Edit 工具更新对应文件
- 在会议报告中注明"已更新 CLAUDE.md"或"已更新 README.md"

### Step 7: 执行判断（自动）
- 含章节号且用户选择"直接写" → 触发Writer执行
- 纯设定讨论 → 保存报告，提示可用 `/novel write X` 执行

## 快速模式

3人5分钟快速讨论：
```
/novel meeting "议题" --quick
```

## 输出

保存到 `06-笔记/会议记录/[议题简称]-会议报告.md`

## 会议优先原则

凡是需要多方讨论的事项，**优先用 `/novel meeting`**，而不是单个Agent判断。

## 附录：完整执行示例

以议题"第50章主角应该输还是赢"为例，展示完整的多轮讨论执行流程：

### Step 1: Editor确认议题
```
Editor读取大纲、人物数据后输出：
- 讨论范围：第50章关键战斗的胜负决策
- 决策类型：二选一（输/赢），但可讨论程度
- 关键考量：主角成长曲线、读者期待、后续剧情铺垫
```

### Step 2: 第一轮 - 并行启动Agent（使用Agent工具）

**同时发起4个Agent调用**：

```
Agent 1: novel-editor
"议题：第50章主角战斗胜负。请输出你的立场和理由..."

Agent 2: novel-reader-proxy
"议题：第50章主角战斗胜负。从读者体验角度分析..."

Agent 3: novel-character-growth-tracker
"议题：第50章主角战斗胜负。评估对主角成长弧光的影响..."

Agent 4: novel-writer
"议题：第50章主角战斗胜负。分析写作可行性和效果..."
```

**收集结果后，主流程汇总**：

```markdown
## 第一轮观点汇总

### @Editor
立场：建议惨胜（赢但付出代价）
核心理由：...

### @Reader-Proxy
立场：希望主角赢，但不要轻松赢
核心理由：...

### @Growth-Tracker
立场：建议输，通过失败促进成长
核心理由：...

### @Writer
立场：两种方式都能写，惨胜更易铺垫后续
核心理由：...
```

### Step 3: 第二轮 - 交叉讨论（再次并行启动Agent）

**再次同时发起4个Agent调用，传入汇总**：

```
Agent 1: novel-editor
"以下第一轮所有Agent的观点汇总：...
请阅读后，对其他Agent的观点进行回应..."

Agent 2: novel-reader-proxy
"以下第一轮所有Agent的观点汇总：...
请阅读后，对其他Agent的观点进行回应..."

Agent 3: novel-character-growth-tracker
"以下第一轮所有Agent的观点汇总：...
请阅读后，对其他Agent的观点进行回应..."

Agent 4: novel-writer
"以下第一轮所有Agent的观点汇总：...
请阅读后，对其他Agent的观点进行回应..."
```

**收集第二轮输出**：

```markdown
## 交叉讨论结果

### @Editor
@Growth-Tracker 关于失败的建议：
【同意】失败确实能促进成长，但...
【补充】建议改为惨胜，既能保留失败的教育意义，又不打击读者

@Reader-Proxy 关于读者期待的反馈：
【同意】读者确实需要爽点，...
```

### Step 4: 如需第三轮

如仍有分歧（如 Growth-Tracker 坚持失败，Editor 坚持惨胜），则进行第三轮。

### Step 5: Editor整合决策

综合所有讨论，输出：
```markdown
## 会议决策报告

### 决议
第50章主角**惨胜**（达成目标但付出重大代价）

### 理由
1. 满足读者爽感需求（Reader-Proxy）
2. 保留成长空间，不破坏战力体系（Growth-Tracker认可）
3. 便于铺垫后续更大危机（Editor）
4. 写作上可通过牺牲队友/受伤体现代价（Writer可行）

### 执行建议
- 战斗过程：主角占据上风 → 反派底牌 → 惨胜
- 代价设计：队友重伤/主角留下隐患
- 章尾钩子：胜利后的新危机浮现
```

### 关键提醒

**必须严格遵循**：
1. 第一轮：并行Agent调用，各自独立输出
2. 主流程汇总（不能跳过）
3. 第二轮：再次并行Agent调用，**必须传入汇总**
4. 不能期望Agent之间直接通信
